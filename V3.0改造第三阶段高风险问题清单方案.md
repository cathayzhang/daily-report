# 开发日报 V3.0 第三阶段：增强交互与扩展性 - 详细设计方案

本文档为"开发日报"项目V3.0开发计划的第三阶段（Phase 3: Interaction & Extensibility）提供详细的技术设计与实施方案。本阶段旨在赋予系统更强的用户交互能力和更灵活的扩展性，核心是实现 **JIRA 问题备注的持久化** 和 **高风险问题清单的动态化**。

---

## **1. 任务3.1: JIRA 问题备注持久化方案**

此功能旨在解决备注无法跨日期留存的问题，实现对重点问题处理进展的持续跟踪。

### **1.1. 总体设计**

我们将引入一个全新的 `remarks_manager.py` 模块，并设计一套以用户侧 CSV 文件为输入、以 SQLite 数据库为存储、以报告侧表格展示为输出的完整数据流。

*   **用户交互**: 用户在项目根目录下维护一个名为 `remarks.csv` 的文件，格式为 `jira_key,remark`。
*   **数据同步**: 程序启动时，`main.py` 将调用 `remarks_manager` 中的函数，读取 `remarks.csv` 的内容，并将其同步（覆盖式更新）到数据库的 `jira_remarks` 表中。
*   **数据查询与展示**: `report_generator.py` 在生成报告时，会从 `remarks_manager` 获取所有备注信息，并在渲染问题列表时，将备注匹配并展示在新增的"备注/日志"列中。

### **1.2. 技术选型与实现细节**

*   **核心技术**: Python, Pandas, sqlite3
*   **模块职责**:
    *   **`database_manager.py`**:
        *   无需新增接口。复用已有的数据库连接和执行SQL的能力。
        *   需确保在 `init_db()` 中已包含创建 `jira_remarks` 表的逻辑，其表结构在总体设计中已定义。
        *   SQL操作将使用 `INSERT OR REPLACE INTO` 语句，以 `jira_key` 为主键，实现备注的更新或插入。
    *   **`remarks_manager.py` (新增)**:
        *   `class RemarksManager`:
            *   `sync_remarks_from_csv(self, db_conn, csv_path='remarks.csv')`:
                *   使用 `pandas.read_csv(csv_path)` 读取文件。
                *   校验文件是否存在及表头是否正确 (`jira_key`, `remark`)。
                *   遍历 DataFrame 的每一行，对每一条备注执行 `INSERT OR REPLACE` SQL语句，写入数据库。
            *   `get_all_remarks(self, db_conn)`:
                *   查询 `jira_remarks` 表中的所有数据。
                *   返回一个以 `jira_key` 为键、`remark` 为值的字典（例如 `{'PROJ-123': 'This is a key issue'}`），供报告生成模块高效查询。
    *   **`report_generator.py` (修改)**:
        *   在生成问题列表前，调用 `remarks_manager.get_all_remarks()` 获取备注字典。
        *   在构建表格的循环中，使用 `remarks_dict.get(issue_key, '')` 的方式安全地获取备注，并将其添加到新列中。
    *   **`main.py` (修改)**:
        *   在主流程的早期，初始化 `RemarksManager` 并调用 `sync_remarks_from_csv()`，完成数据同步。

### **1.3. 数据流**

`remarks.csv` -> `main.py` -> `RemarksManager.sync_remarks_from_csv()` -> `DatabaseManager` -> `dev_report.db` (写入)

`dev_report.db` (读取) -> `RemarksManager.get_all_remarks()` -> `ReportGenerator` -> `report.html` (展示)

## **2. 任务3.2 & 3.3: 高风险清单动态化方案**

此功能旨在将高风险问题的筛选逻辑从代码中解耦，允许用户通过配置文件灵活定义筛选规则。

### **2.1. 总体设计**

我们将通过扩展 `config.ini` 文件，让用户能够自定义一系列筛选规则。这些规则由 `config_loader.py` 解析后，传递给 `analyzer.py`，后者将动态地应用这些规则在 Pandas DataFrame 上进行数据过滤。

### **2.2. 技术选型与实现细节**

*   **核心技术**: Python, configparser, Pandas
*   **模块职责**:
    *   **`config.ini` (修改)**:
        *   新增 `[HighRiskRules]` section。
        *   **规则格式**: `key = column_name, operator, value`
        *   `key` 是任意唯一的规则名，例如 `rule1`, `rule2`。
        *   `column_name` 是 JIRA 数据中的列名（如 `priority`, `status`）。
        *   `operator` 是预定义的比较操作符。
        *   `value` 是用于比较的值。
    *   **`config_loader.py` (修改)**:
        *   `class ConfigLoader`:
            *   新增方法 `get_high_risk_rules(self)`:
                *   使用 `configparser` 读取 `[HighRiskRules]` section。
                *   遍历所有规则，将每条规则（`column,operator,value`）解析并转换为一个结构化的字典。
                *   对于需要列表作为值的操作符（如 `in`, `not_in`），将逗号分隔的字符串值转换为 Python 列表。
                *   返回一个规则字典的列表，例如 `[{'column': 'status', 'operator': 'not_in', 'value': ['Closed', 'Resolved']}, ...]`。
                *   包含错误处理机制，例如当规则格式不正确时，打印警告并跳过该规则。
    *   **`analyzer.py` (修改)**:
        *   `class Analyzer`:
            *   重构 `get_high_risk_issues` 方法，使其接收规则列表作为参数：`get_high_risk_issues(self, jira_df, rules)`。
            *   方法内部实现：
                1.  初始化一个全为 `True` 的 Pandas Series 作为基础布尔掩码（`final_mask`）。
                2.  遍历 `rules` 列表中的每一条规则字典。
                3.  根据规则的 `operator` 构建一个临时的布尔掩码。例如，如果 `operator` 是 `greater_than`，则构建 `pd.to_numeric(jira_df[column]) > float(value)`。
                4.  使用逻辑与（`&`）将临时掩码与 `final_mask` 合并：`final_mask = final_mask & temporary_mask`。
                5.  循环结束后，使用 `final_mask` 过滤 DataFrame: `return jira_df[final_mask]`。

### **2.3. 规则定义与示例**

为了确保灵活性和易用性，我们定义以下操作符：

| 操作符 (`operator`) | 描述 | `value` 格式示例 | `config.ini` 示例 |
| :--- | :--- | :--- | :--- |
| `equals` | 等于 | 单个值 | `rule1 = priority, equals, High` |
| `not_equals` | 不等于 | 单个值 | `rule2 = status, not_equals, Done` |
| `contains` | 字符串包含 | 单个字符串 | `rule3 = labels, contains, tech-debt` |
| `in` | 属于列表之一 | 逗号分隔的值 | `rule4 = priority, in, Highest,High` |
| `not_in` | 不属于列表 | 逗号分隔的值 | `rule5 = status, not_in, Closed,Resolved` |
| `greater_than` | 数值大于 | 一个数字 | `rule6 = story_points, greater_than, 8` |
| `less_than` | 数值小于 | 一个数字 | `rule7 = di_score, less_than, 10` |

**注意**: 对于数值比较，`analyzer.py` 内部将使用 `pd.to_numeric(series, errors='coerce')` 来强制转换数据类型，以避免因数据格式问题导致程序崩溃。

## **3. 实施总结**

第三阶段的开发将显著提升系统的实用性和灵活性。开发完成后，用户将能够：
1.  通过维护一个简单的 CSV 文件，实现对关键问题备注的永久记录和跟踪。
2.  通过修改配置文件，无需改动任何代码，即可完全自定义高风险问题的筛选标准，适应不同项目阶段的管理需求。

后续的 **第四阶段** 将在此基础上，对报告UI进行相应优化，并更新用户手册，以确保所有新功能都能被用户充分理解和使用。
