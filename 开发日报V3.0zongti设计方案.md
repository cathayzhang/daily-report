# 开发日报 V3.0 总体设计与开发计划

本文档包含了"开发日报"项目V3.0版本的总体设计方案和详细的迭代开发计划，旨在为接下来的开发工作提供清晰、全面的指导。

---

## **第一部分：总体设计方案**

### **1. V3.0 总体技术架构建议**

V3.0 的核心架构升级是从一个无状态的单次分析工具，演进为一个**有状态的、具备历史数据持久化能力的本地化分析平台**。

*   **核心变革：引入SQLite数据持久化层**
    我们将彻底替换当前基于`history.csv`的方案，引入一个轻量级的 **SQLite 数据库**作为系统的统一数据后端。这个数据库将以单个文件（例如 `dev_report.db`）的形式存储在项目目录中。
*   **为什么选择SQLite?**
    *   **完全本地化，确保信息安全**: SQLite是无服务器的本地文件数据库，所有操作均在本地完成，不涉及任何网络请求。这从根本上满足了您"断网环境、数据不出本地"的最高安全要求。
    *   **强大的数据管理**: 相比CSV或JSON，SQLite能更好地管理结构化关系数据，为历史KPI、问题备注、动态配置等复杂数据提供稳定、高效的存储和查询能力。
    *   **轻量且内建**: Python原生支持（通过`sqlite3`模块），无需安装和配置额外的数据库服务，保持了项目的轻量级特性。

*   **新模块与职责划分**:
    *   `database_manager.py` (新增): 一个全新的底层模块，封装所有对SQLite数据库的连接、建表和CRUD（增删改查）操作，实现数据库逻辑的集中管理。
    *   `history_manager.py` (重构): 将不再直接操作文件，而是调用`database_manager.py`，负责将每日的KPI数据写入数据库。
    *   `remarks_manager.py` (新增): 一个新的业务模块，负责调用`database_manager.py`来管理JIRA问题的备注信息。

### **2. 各核心需求技术方案详述**

#### **需求1: 历史数据与趋势分析**

*   **问题分析**: 无法进行跨时间周期的趋势分析。
*   **技术方案**:
    1.  在SQLite数据库中创建`daily_kpis`表，用于存储每日的核心指标。
    2.  重构`history_manager.py`，将当日的分析结果（问题总数、各级问题数、DI值等）写入`daily_kpis`表。
    3.  改造`visualizer.py`，使其从数据库中读取历史数据来生成所有趋势图表。
*   **涉及模块变更**: `history_manager.py`, `visualizer.py`, `main.py`, `analyzer.py`。
*   **新增模块**: `database_manager.py`。
*   **数据结构/接口定义 (Data Schema)**:
    ```sql
    -- 表: daily_kpis
    CREATE TABLE daily_kpis (
        report_date DATE PRIMARY KEY,
        total_issues INTEGER,
        a_issues INTEGER,
        b_issues INTEGER,
        c_issues INTEGER,
        di_value REAL
    );
    ```

#### **需求2: 燃尽图数据自动化计算**

*   **问题分析**: DI燃尽图数据依赖手动提供，易出错且与实际数据不一致。
*   **技术方案**:
    1.  将项目的计划数据（起止日期、起止DI值）配置在`config.ini`文件的`[BurndownPlan]`部分。
    2.  每日实际DI值由`analyzer.py`根据当日导入数据自动计算，并由`history_manager.py`存入数据库。
    3.  `visualizer.py`从数据库获取全部历史DI实际值，从配置文件获取计划，动态生成"实际"与"理想"两条曲线的燃尽图。
*   **涉及模块变更**: `config.ini`, `config_loader.py`, `analyzer.py`, `visualizer.py`。
*   **数据结构/接口定义 (Data Schema)**:
    ```ini
    ; config.ini
    [BurndownPlan]
    start_date = 2023-01-01
    end_date = 2023-03-31
    start_di = 300
    target_di = 20
    ```

#### **需求3: 高风险问题备注的持久化与映射**

*   **问题分析**: 手动添加的备注在第二天会丢失，无法连续跟踪。
*   **技术方案**:
    1.  在SQLite中创建`jira_remarks`表，以JIRA Key为主键，存储备注文本和更新时间。
    2.  新增`remarks_manager.py`模块，封装对`jira_remarks`表的增、改、查逻辑。
    3.  用户通过维护一个本地的`remarks.csv`文件（包含`jira_key`, `remark`两列）来批量导入或更新备注。
    4.  `report_generator.py`在生成报告时，会遍历问题清单，通过`remarks_manager.py`查询并嵌入对应备注。
*   **涉及模块变更**: `report_generator.py`, `main.py`。
*   **新增模块**: `remarks_manager.py`。
*   **数据结构/接口定义 (Data Schema)**:
    ```sql
    -- 表: jira_remarks
    CREATE TABLE jira_remarks (
        jira_key TEXT PRIMARY KEY,
        remarks TEXT,
        last_updated TIMESTAMP
    );
    ```

#### **需求4: 高风险问题清单的动态定制**

*   **问题分析**: 高风险清单的筛选维度硬编码，不够灵活。
*   **技术方案**:
    1.  在`config.ini`中新增`[HighRiskRules]`部分，允许用户以`列名, 操作符, 值`的格式自定义多条筛选规则。
    2.  `config_loader.py`负责解析这些规则。
    3.  重构`analyzer.py`的核心筛选函数，使其接收规则列表作为参数，并动态地应用这些规则到DataFrame上进行过滤。
*   **涉及模块变更**: `config.ini`, `config_loader.py`, `analyzer.py`。
*   **数据结构/接口定义 (Data Schema)**:
    ```ini
    ; config.ini
    [HighRiskRules]
    rule1 = priority, equals, High
    rule2 = status, not_in, ['Closed', 'Resolved']
    rule3 = labels, contains, on-fire
    ```

#### **需求5: "收敛偏差"指标的明确化**

*   **问题分析**: 指标定义、计算和展示方式不明确。
*   **技术方案**:
    *   **定义**: "收敛偏差" = "理想DI存量" - "实际DI存量"。它衡量问题收敛速度与计划的偏离程度。正值表示领先，负值表示落后。
    *   **计算逻辑**: 在`analyzer.py`中实现。理想DI根据燃尽图计划线性计算，实际DI从数据库中获取。
    *   **数据可视化**: 在`visualizer.py`中创建一个新的图表函数。建议使用**带面积填充的柱状图**，在报告中新增"收敛偏差趋势"图，用不同颜色（如蓝/红）表示领先和落后。
*   **涉及模块变更**: `analyzer.py`, `visualizer.py`, `report_generator.py`, `templates/report_template.html`。

### **3. 分阶段实施计划 (总结)**

1.  **第一阶段：奠定数据基石**。搭建SQLite数据库及`database_manager.py`，重构`history_manager.py`，完成**需求1**。
2.  **第二阶段：自动化核心图表**。实现燃尽图自动化（**需求2**）和收敛偏差指标（**需求5**）。
3.  **第三阶段：增强交互与扩展性**。实现备注持久化（**需求3**）和高风险清单动态化（**需求4**）。
4.  **第四阶段：UI/UX优化与文档更新**。调整报告模板，完善用户手册。

    *   为"燃尽图"和"收敛偏差图"增加新的图表容器。

    *   高风险清单的标题应能动态反映当前的筛选规则。

*   **文档**:

## **第二部分：迭代开发计划**

#### **第一阶段：奠定数据基石 (Phase 1: Data Foundation)**
*此阶段的目标是完成从CSV到SQLite的架构迁移，为所有新功能打下坚实基础。*

*   **任务1.1: 创建并实现 `database_manager.py`**
    *   **描述**: 创建全新的 `database_manager.py` 模块。此模块将封装所有与SQLite的底层交互，包括：获取数据库连接、执行SQL语句、以及在程序首次运行时初始化数据库和所有表结构（`daily_kpis`, `jira_remarks`等）。这是整个持久化层的基石。
    *   **复杂度**: 7

*   **任务1.2: 重构 `history_manager.py` 以使用数据库**
    *   **描述**: 重构 `history_manager.py`。移除所有读写CSV文件的代码，改为调用 `database_manager.py` 提供的接口。核心功能是接收 `analyzer.py` 计算出的当日KPI数据，并将其作为一条新记录插入到 `daily_kpis` 表中。
    *   **复杂度**: 6

*   **任务1.3: 集成并验证历史趋势分析功能**
    *   **描述**: 调整 `main.py`, `analyzer.py`, 和 `visualizer.py`。确保 `main.py` 能正确调用新的 `history_manager`。`visualizer.py` 的数据源需要从读取CSV文件改为从 `daily_kpis` 表查询数据，以正确生成历史趋势图。此任务的目标是让原有的核心功能在新的数据架构下恢复正常。
    *   **复杂度**: 5

---

#### **第二阶段：自动化核心图表 (Phase 2: Automated Charts)**
*此阶段专注于自动化生成关键业务图表，提升报告的数据一致性和深度。*

*   **任务2.1: 实现燃尽图计划的配置化**
    *   **描述**: 在 `config.ini` 文件中新增 `[BurndownPlan]` 部分。相应地，在 `config_loader.py` 中增加解析此部分的代码，读取项目的开始/结束日期以及起始/目标DI值，并将其提供给其他模块使用。
    *   **复杂度**: 3

*   **任务2.2: 实现燃尽图的自动化生成**
    *   **描述**: 重构 `visualizer.py` 中的燃尽图生成函数。该函数需从配置中获取计划数据以计算"理想线"，并从数据库`daily_kpis`表中查询所有历史DI值以绘制"实际线"，最终将两条线在同一图表中展示。
    *   **复杂度**: 6

*   **任务2.3: 开发"收敛偏差"指标的计算与可视化**
    *   **描述**: 实现"收敛偏差"指标。这需要两部分工作：1. 在 `analyzer.py` 中新增一个函数，根据燃尽图计划和每日实际DI，计算出每日的"收敛偏差"值。2. 在 `visualizer.py` 中新增一个函数，将计算出的偏差序列数据，渲染成一个带有正负颜色区分的柱状图。
    *   **复杂度**: 7

---

#### **第三阶段：增强交互与扩展性 (Phase 3: Interaction & Extensibility)**
*此阶段将赋予用户更大的灵活性和自定义能力。*

*   **任务3.1: 创建 `remarks_manager.py` 并实现备注持久化**
    *   **描述**: 创建全新的 `remarks_manager.py` 模块。该模块负责处理所有与JIRA备注相关的业务逻辑，包括：从约定的CSV文件读取备注、调用`database_manager`将备注写入`jira_remarks`表、以及根据JIRA Key提供查询备注的接口供`report_generator`调用。
    *   **复杂度**: 7

*   **任务3.2: 实现高风险清单的动态规则配置**
    *   **描述**: 在 `config.ini` 中新增 `[HighRiskRules]` 部分，并更新 `config_loader.py` 以支持解析这些自定义规则。每条规则应被解析成结构化数据（如字典），包含`列名`, `操作符`, `值`等关键信息。
    *   **复杂度**: 4

*   **任务3.3: 重构 `analyzer.py` 以支持动态筛选逻辑**
    *   **描述**: 重构 `analyzer.py` 中的高风险问题筛选函数。使其不再使用任何硬编码的逻辑，而是接收一个由`config_loader`生成的规则列表。函数内部需要根据这个规则列表动态构建Pandas查询语句或布尔索引，以实现完全由配置驱动的动态筛选。
    *   **复杂度**: 8

---
