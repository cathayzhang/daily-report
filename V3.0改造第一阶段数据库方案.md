# V3.0 改造第一阶段：数据库方案设计

本文档是 "开发日报 V3.0" 项目第一阶段 "奠定数据基石" 的详细技术设计与实施方案，旨在指导从 CSV 文件存储到 SQLite 数据库的架构迁移。

---

## 1. 核心目标与范围

**核心目标**: 将项目的核心数据持久化机制从不稳定的 `history.csv` 文件，全面迁移到一个轻量、可靠的 **SQLite 数据库**。

**本阶段范围**:
- **构建**一个新的数据库管理模块，用于封装所有底层数据库交互。
- **重构**现有的历史数据管理模块，使其通过新的数据库模块来存取数据。
- **集成**并**验证**，确保所有依赖历史数据的功能（如趋势图）在新架构下能正确工作。
- **不涉及**任何新功能的开发，仅做底层重构。

---

## 2. 技术选型

- **数据库**: **SQLite**
- **驱动**: Python 内置的 `sqlite3` 模块。

**选型理由**:
- **零依赖**: 无需安装和配置外部数据库服务，保持项目轻量化。
- **完全本地化**: 数据库以单个文件形式 (`dev_report.db`) 存在于项目目录，所有数据和操作都在本地完成，满足项目的核心安全要求。
- **事务性支持**: 提供 ACID 事务能力，确保数据写入的原子性和一致性，远优于CSV。

---

## 3. 数据库表结构 (Schema)

在数据库 `dev_report.db` 中，我们将初始化以下两张核心数据表。

#### 3.1 `daily_kpis` (每日核心指标表)
用于存储每日生成的关键绩效指标，是趋势分析的数据来源。

```sql
CREATE TABLE IF NOT EXISTS daily_kpis (
    report_date TEXT PRIMARY KEY, -- 报告日期 (格式: 'YYYY-MM-DD')，作为唯一标识
    total_issues INTEGER,       -- 当日JIRA问题总数
    a_issues INTEGER,           -- 当日A级问题数
    b_issues INTEGER,           -- 当日B级问题数
    c_issues INTEGER,           -- 当日C级问题数
    di_value REAL               -- 当日计算出的DI（缺陷指数）值
);
```

#### 3.2 `jira_remarks` (JIRA 问题备注表)
用于持久化存储对特定 JIRA 问题的备注信息。此表的功能将在后续阶段完全实现，但表结构在此阶段预先创建。

```sql
CREATE TABLE IF NOT EXISTS jira_remarks (
    jira_key TEXT PRIMARY KEY,    -- JIRA 问题唯一标识 (例如: 'PROJ-123')
    remarks TEXT,                 -- 备注的具体内容
    last_updated TEXT             -- 备注最后更新时间 (格式: ISO 8601)
);
```

## 4. 核心模块设计与实现

#### 4.1. **新增**: `database_manager.py`
这是全新的数据库底层访问模块，是整个数据持久化方案的基石。

- **职责**:
    1.  管理数据库连接。
    2.  在程序首次运行时，自动创建数据库文件及上述数据表。
    3.  提供通用的、参数化的数据读写接口，防止SQL注入。

- **建议实现 (伪代码)**:
  ```python
  import sqlite3
  import pandas as pd

  DB_FILE = 'dev_report.db'

  def init_db():
      # ... 创建上述两张表的 SQL ...
      with sqlite3.connect(DB_FILE) as conn:
          cursor = conn.cursor()
          cursor.execute(CREATE_DAILY_KPIS_TABLE_SQL)
          cursor.execute(CREATE_JIRA_REMARKS_TABLE_SQL)
      print("数据库初始化完成。")

  def execute_update(query, params=()):
      with sqlite3.connect(DB_FILE) as conn:
          cursor = conn.cursor()
          cursor.execute(query, params)
          conn.commit()

  def query_to_dataframe(query, params=()):
      with sqlite3.connect(DB_FILE) as conn:
          df = pd.read_sql_query(query, conn, params=params)
      return df
  ```

#### 4.2. **重构**: `history_manager.py`
此模块将从文件操作者转变为数据库业务逻辑的封装者。

- **职责**: 专注于KPI历史数据的业务逻辑，调用 `database_manager` 完成实际存储。

- **改造方案**:
    1.  **移除**所有 `pandas.read_csv()` 和 `to_csv()` 相关代码。
    2.  **引入** `import database_manager`。
    3.  **修改/创建**以下函数:
        - `save_kpis(report_date, kpis_dict)`:
            - **逻辑**: 将传入的KPI字典构建成 `INSERT OR REPLACE` 语句。`OR REPLACE` 的使用至关重要，它能根据主键 `report_date` 自动处理记录的插入或更新，保证了数据幂等性。
            - **调用**: `database_manager.execute_update(sql, params)`
        - `load_kpi_history()`:
            - **逻辑**: 构建 `SELECT * FROM daily_kpis ORDER BY report_date ASC` 查询。
            - **调用**: `database_manager.query_to_dataframe(sql)`，将查询结果直接作为 DataFrame 返回。

## 5. 集成与验证计划

#### 5.1. **模块集成**

- **`main.py`**:
    1.  在程序启动的入口处，增加对 `database_manager.init_db()` 的调用，确保数据库和表结构始终存在。
    2.  在每日分析流程的末尾，调用 `history_manager.save_kpis()` 将当天的分析结果存入数据库。
- **`visualizer.py`**:
    1.  找到所有绘制历史趋势图的函数。
    2.  将其数据源获取方式，从读取CSV文件，改为调用 `history_manager.load_kpi_history()` 来获取包含所有历史数据的DataFrame。

#### 5.2. **验证步骤清单**

为确保迁移成功，请按以下步骤进行严格测试：

1.  **环境清理**: 备份并删除旧的 `history.csv` 文件和 `dev_report.db` 文件（如果存在）。
2.  **首次运行**:
    - **执行**: 运行主程序 `main.py`。
    - **验证**:
        - `dev_report.db` 文件被成功创建。
        - 使用DB Browser等工具检查 `daily_kpis` 表，应包含 **1** 条当天日期的记录。
        - 查看生成的HTML报告，历史趋势图应正确显示 **1** 个数据点。
3.  **二次运行 (模拟次日)**:
    - **准备**: 修改输入数据，模拟第二天的情况。
    - **执行**: 再次运行主程序 `main.py`。
    - **验证**:
        - `daily_kpis` 表应包含 **2** 条记录。
        - 新生成的HTML报告中，历史趋势图应正确、连续地显示 **2** 个数据点。
4.  **幂等性测试**:
    - **执行**: 使用与第一次运行完全相同的数据，再次运行主程序。
    - **验证**:
        - `daily_kpis` 表应仍然只包含 **2** 条记录（或取决于你的模拟数据天数），当天的数据被更新而不是新增。
        - 检查对应日期的记录值是否已更新（如果数据有变）。

通过以上所有验证点，即可确认第一阶段的架构迁移成功完成。
